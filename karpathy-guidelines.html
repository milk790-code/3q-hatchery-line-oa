01

最小可用

只寫解決當前問題所需的最少程式碼。三行相似比一個過早的抽象更好。不要設計你還不需要的功能。

  • 這個功能現在真的需要嗎？
  • 有沒有更簡單的解法？
  • 抽象層次是否必要，還是只是方便？

02

程式碼即說明

良好命名的函式和變數本身就是文件。不寫解釋「做什麼」的注釋，只寫「為什麼」——非顯而易見的約束或 workaround。

  • 函式名稱能清楚說明它做什麼嗎？
  • 注釋是否說明了「為什麼」而非「是什麼」？
  • 刪掉注釋後，未來的讀者還能理解嗎？

03

不當英雄

遇到瓶頸時，先嘗試最簡單的解法。不要花三天做可以用 cron job 解決的問題。優雅感是結果，不是目標。

  • 有沒有現成的工具或 API 可以用？
  • 手動解決一次是否比自動化更划算？
  • 技術債的代價是否清楚列出？

04

資料優先

先理解資料流，再設計架構。Schema 和 API 契約比框架選擇更重要。壞的資料模型會讓好的程式碼變得複雜。

  • 資料從哪裡來，要到哪裡去？
  • Schema 是否反映了真實的業務邏輯？
  • API 契約是否明確，邊界條件是否處理？

05

可讀性 > 聰明

未來的你（或 AI）要能快速理解。一個清楚的 if/else 比一個聰明的 one-liner 更有價值。程式碼讀的次數遠多於寫的次數。

  • 三個月後的你能看懂這段嗎？
  • 可以拆成更小的具名函式嗎？
  • 魔法數字是否都有常數名稱？

06

邊界驗證

只在系統邊界（用戶輸入、外部 API）做驗證。不要為內部程式碼加防禦性錯誤處理。信任你控制的程式碼。

  • 驗證是否在正確的層級（外部邊界）？
  • 有沒有多餘的 try/catch 包著內部邏輯？
  • 錯誤訊息對用戶有意義嗎？

07

可逆性優先

優先選擇容易回退的方案。資料庫 migration 要可回滾，部署要有 canary 或 rollback 指令。不要用破壞性操作解決問題。

  • 這個操作可以回滾嗎？
  • 影響範圍是否最小化？
  • 出事時，有沒有明確的回復步驟？

08

小步提交

每個 commit 只做一件事，且能描述為一句話。Git history 是最好的文件，也是最快的除錯工具。

  • 這個 commit message 能一句話說清楚嗎？
  • 有沒有混入不相關的變更？
  • 測試和程式碼是否在同一個 commit？

3Q Hatchery 專屬補充

Cloudflare Workers

有 1ms CPU 限制，避免同步的 heavy computation。善用 `ctx.waitUntil()` 處理非同步任務。

D1 查詢

優先使用 prepared statements，避免 N+1 問題。每次 Worker 調用控制查詢次數 ≤ 5。

KV 存取

Batch 讀取，避免每個 key 一次 await。用 `getWithMetadata` 同時讀取值和過期時間。

設計系統

HTML/CSS 直接用 `colors_and_type.css` 的 token，不要硬寫顏色值。黃金比例：金色 ≤ 3%。
